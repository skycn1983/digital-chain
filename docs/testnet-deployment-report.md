### 2. 链同步与一致性

**测试步骤**:
1. 启动所有节点（初始高度=1，创世块）
2. 节点1挖矿到高度8（7个新区块）
3. 等待P2P同步
4. 验证所有节点链高度

**结果**: ⚠️ 部分成功

| 节点 | 初始高度 | 目标高度 | 实际高度 | 状态 |
|------|---------|---------|---------|------|
| Node1 | 1 | 8 | 8 | ✅ |
| Node2 | 1 | 8 | 1→8 (断开重连后) | ⚠️ |
| Node3 | 1 | 8 | 1 | ❌ |

**关键发现**:
- 初始 handshake 后，节点2/3未能自动同步到节点1的链
- **断开重连**后，同步成功（节点2验证通过）
- 同步机制本身有效，但初始连接时序有问题

**问题根因**:
节点2/3 handshake 时，节点1链高度=1（未挖矿），节点2/3认为同步非必需（虽然条件已改为 `>=`，但可能 handshake 响应中节点1报告的高度为1，`>=` 成立，但 `_requestSync` 未被调用？）

实际上，节点2/3 handshake 后应该请求同步，但可能因为：
1. 请求发送但节点1响应慢或丢失
2. 节点2/3处理 blocks 时因创世块不匹配触发 needsFullSync，但后续处理中断

**解决方案已验证**:
- 手动断开重连触发同步 ✅
- 等待更长时间可能自动同步（未验证）
- 改进：在 handshake 后强制请求完整链（如果高度相同）

### 3. 网络自愈能力

**测试**: 断开节点2连接，观察重连和同步

**步骤**:
1. 节点2与节点1断开（POST /network/disconnect）
2. 节点2自动重连（重连队列机制）
3. 验证链同步

**结果**: ✅ 成功

```
断开节点2 → 节点2在5秒内尝试重连 → 重连成功 → 链同步到高度8
```

**观察**:
- 重连机制工作正常
- 重连后自动触发同步
- 链状态恢复一致

### 4. 性能基准（单节点）

由于多节点同步问题尚未完全解决，先提供单节点基准：

**测试环境**: Node1 单节点，无P2P
**配置**: 并发10，总交易100

| 指标 | 数值 |
|------|------|
| TPS | 21.74 |
| 平均延迟 | 11 ms |
| p95 延迟 | 40 ms |
| p99 延迟 | 40 ms |
| 最大延迟 | 40 ms |

**问题**: 并发 nonce 冲突导致19/100交易失败（同一钱包重复nonce）

**优化建议**:
- 为每个请求分配独立 nonce
- 或使用多钱包并发

## 🔧 发现的问题与修复

### 问题1: 创世块不一致导致无法合并

**现象**: 节点使用不同数据目录，各自创建不同创世块，链无法合并。

**修复**:
1. 修改 `_requestSync` 逻辑：新节点（本地高度=1）请求完整链（fromHeight=0）
2. 添加创世块替换逻辑：当收到 index=0 的区块且 hash 不同时，直接替换链
3. 修改同步条件为 `>=`，覆盖相同高度场景

**文件**: `src/p2p/server.js`, `src/p2p/sync/fork-choice.js`

---

### 问题2: 初始 handshake 同步时机

**现象**: 节点2/3 handshake 时，节点1高度=1，请求同步后未能持续同步到后续区块。

**临时解决**: 手动断开重连触发同步

**待优化**:
- handshake 后如果链高度相同，延迟1秒再次检查并请求同步
- 或者，收到新区块广播时，如果本地高度低于广播者，主动请求增量同步

---

### 问题3: 交易 nonce 冲突

**现象**: 并发请求使用同一 nonce 导致交易被拒绝

**影响**: 性能测试失败率 95%

**修复**: 修改 benchmark 脚本为每个请求分配独立 nonce（或串行化）
